klar.gg sells time-bound software licenses. Somebody pays an invoice, and within a few seconds they get an email with a key, and the key shows up on their purchase page in the forum, and one click copies it to their clipboard. None of that involves staff intervention.

Three services make that happen. None of them know much about the others.

1. **IPS Nexus** runs the storefront, processes the invoice, and fires a `markPaid` event.
2. **An IPS plugin** (`klar.gg panel`) hooks that event, asks an API for a fresh key, and saves it back to the order.
3. **Overwatch**, a FastAPI service, is the API. It does not own any data. It logs into a legacy PHP admin panel, scrapes the HTML, and returns clean JSON.

The interesting bits are at the seams between them.

## Why Overwatch exists at all

The licensing system is older than the storefront. It is a PHP web-app at `admin.klar.ac` that issues, tracks, and validates license keys. There is no JSON API. To do anything programmatically you have to:

1. POST credentials to `/login.php` and capture a session cookie.
2. Make a follow-up request, get back an HTML page, and scrape table rows out of it.

That works fine when a human is clicking buttons. It is a non-starter when an IPS plugin needs to ask "give me three keys for product 9 with a 720-hour duration" the moment an invoice clears.

So Overwatch sits in front of it. The plugin talks to Overwatch. Overwatch talks to the legacy panel. The plugin never sees an HTML page.

## Wrapping the panel with FastAPI

Auth is the first problem. Each call from the plugin should go through the legacy panel as the configured staff user, but spinning up a new login on every request is slow and noisy. The middleware uses HTTP Basic on the Overwatch side, then translates that into a cached session cookie for the legacy panel:

```python
PANEL_BASE_URL = "https://admin.klar.ac"
SESSION_CACHE_PREFIX = "overwatch:panel_session:"
SESSION_CACHE_TTL_SECONDS = 15 * 60

async def panel_auth_middleware(request, call_next):
    auth_header = request.headers.get("Authorization")
    if not auth_header or not auth_header.startswith("Basic "):
        return _unauthorized("Unauthorized")

    decoded = base64.b64decode(auth_header[6:]).decode("utf-8")
    username, password = decoded.split(":", 1)

    session = aiohttp.ClientSession(cookie_jar=aiohttp.CookieJar())
    cached = await _load_cached_session(username, password)
    if cached:
        session.cookie_jar.update_cookies(cached, response_url=URL(PANEL_BASE_URL))
    else:
        async with session.post(
            f"{PANEL_BASE_URL}/login.php",
            data={"username": username, "password": password},
            allow_redirects=False,
        ) as res:
            if res.status != 302:
                await session.close()
                return _unauthorized()
        await _save_cached_session(username, password, _extract_cookies(session.cookie_jar))

    request.state.panel_session = session
    return await call_next(request)
```

A successful login on the legacy panel responds with a 302 redirect; that is the signal we use to know the credentials worked. The Redis cache stores the cookie jar keyed by username, with the password hashed via `hmac.compare_digest` to detect rotation. The TTL is 15 minutes, which is shorter than the legacy panel's session lifetime and keeps stale sessions from piling up.

The actual endpoints scrape HTML. Here is the abridged user-search route:

```python
@router.get("/users/{username}", response_model=Envelope[UserInfo])
async def search_user(request, response, username):
    session = _session(request)
    async with session.get(
        f"{PANEL_BASE_URL}/users.php?search={username.lower()}"
    ) as res:
        html = await res.text()
        soup = BeautifulSoup(html, "html.parser")

        row = next(
            (tr.find_all("td") for tr in soup.find_all("tr")
             if len(tr.find_all("td")) >= 2
             and username.lower() in tr.find_all("td")[1].get_text(strip=True).lower()),
            None,
        )
        if not row or len(row) < 3:
            response.status_code = 404
            return Envelope[UserInfo](status=404, error="User not found")

        return Envelope[UserInfo](
            status=200,
            data=UserInfo(
                id=int(row[0].get_text(strip=True)),
                user=row[1].get_text(strip=True),
                subscriptions=_parse_subs(row[2].get_text(strip=True)),
            ),
        )
```

Every endpoint follows the same shape: hit an endpoint on the legacy panel, parse the response, return a typed Pydantic envelope. The plugin (and the Discord bot, and anything else we build later) only ever sees the clean JSON.

## What the plugin does on `markPaid`

The IPS hook lives at `plugins/panel/hooks/Invoice.php`. The interesting part is what happens after `parent::markPaid()` returns:

```php
foreach ( $this->items as $item )
{
    if ( !( $item instanceof \IPS\nexus\Invoice\Item\Purchase ) ) continue;

    $packageId = $item->id;
    $panelProductName = $productMap[ (string) $packageId ] ?? NULL;
    if ( !$panelProductName ) continue;

    // Read the duration the customer chose at checkout.
    $durationHours = $this->_resolveDuration( $item, $packageId );
    if ( $durationHours === NULL ) continue;

    // Fetch product list from Overwatch (cached per invoice).
    if ( $panelProducts === NULL ) {
        $panelProducts = $this->_fetchPanelProducts( $apiUrl, $username, $password );
    }
    $panelProductId = $panelProducts[ $panelProductName ] ?? NULL;
    if ( !$panelProductId ) continue;

    // Find purchase rows that don't have a license key yet.
    $rowsNeedingKey = $this->_unkeyedPurchaseRows( $packageId );
    if ( !$rowsNeedingKey ) continue;

    // Ask Overwatch for one fresh key per purchase row.
    $keys = \IPS\Http\Url::external( $apiUrl . '/keys' )
        ->request( 15 )
        ->login( $username, $password )
        ->setHeaders( array( 'Content-Type' => 'application/json' ) )
        ->post( json_encode( array(
            'product_id' => $panelProductId,
            'duration'   => $durationHours,
            'count'      => count( $rowsNeedingKey ),
        ) ) )
        ->decodeJson();

    // Save each key against its purchase, then email the customer.
    foreach ( $rowsNeedingKey as $i => $row ) {
        $purchase = \IPS\nexus\Purchase::constructFromData( $row );
        $lk = new \IPS\nexus\Purchase\LicenseKey\Standard;
        $lk->purchase = $purchase;
        $lk->key = $keys['data']['keys'][ $i ];
        $lk->save();

        $this->_sendDeliveryEmail( $purchase, $lk->key );
    }
}
```

A few things this gets right that took iteration.

**The product map lives in plugin settings.** Each IPS package stores the panel product name it corresponds to (`{ "9": "Fortnite", "12": "Rust" }`). The plugin uses Overwatch's `/products` endpoint at runtime to translate the name into a panel product ID. This means panel product IDs can be re-numbered on the legacy side without breaking the plugin.

**The duration comes from a custom field type the plugin defines.** A package has a Duration field, configured with options like `30 Days:720` or `90 Days:2160`. The customer picks one at checkout. At paid time the plugin parses the colon-separated value and sends the integer hours to Overwatch.

**Idempotency is per purchase row, not per invoice.** Reprocessing an invoice will only generate keys for purchase rows that do not already have one. If Overwatch returns three keys but only two purchase rows are unkeyed, the third key never gets used.

**Errors fail loudly into a custom log channel** (`klar_panel`), which surfaces inside AdminCP. If the panel's HTML structure shifts and Overwatch returns 502s, the staff sees it before customers do.

## The last 2 inches: the theme

By the time the customer visits their purchase page, the key is already saved. The theme's only job is to make it copy in one click. From `themes/2/html/nexus/front/clients/purchase.phtml`:

```phtml
{{if $lkey = $purchase->licenseKey()}}
    <div class='ipsAreaBackground_light ipsPad ipsType_center ipsSpacer_bottom'>
        <h2 class='ipsType_minorHeading'>{lang="purchase_license_key"}</h2>
        <div class='cCopyable' id='licenseKey'
             data-clipboard-text='{$lkey->key}'>
            <i class='fa fa-clone cCopyable_icon' aria-hidden='true'></i>
            <span class='cCopyable_value ipsType_monospace cNexusLicenseKey'>
                {$lkey->key}
            </span>
        </div>
    </div>
{{endif}}
```

The `.cCopyable` class is wired up by an eleven-line vanilla JS handler in the theme's `globalTemplate.phtml` (covered in [the theme post](/blog/reskinning-ips-without-forking)). One click, key on the clipboard, a green border flash for 900ms.

## What this taught me

The pipeline is three services, but each one has a small, defensible job:

- **IPS** owns the customer (auth, checkout, billing, email infrastructure).
- **Overwatch** owns the access to the legacy panel (login flow, session caching, HTML scraping). Anything else we build later, say a Discord slash command to look up a license key, calls the same endpoints.
- **The plugin** owns the translation between IPS concepts (packages, invoices, purchase rows) and panel concepts (products, durations, keys).
- **The theme** owns the surface area the customer sees.

Each piece is small enough that swapping one is plausible. If the legacy panel ever ships a real JSON API, Overwatch becomes a thin pass-through and nothing else changes. If we move the storefront off IPS, the plugin goes away but Overwatch and the theme do not have to.

That is the whole reason for the layering. Not architecture for its own sake, but so each thing can change at its own pace.
